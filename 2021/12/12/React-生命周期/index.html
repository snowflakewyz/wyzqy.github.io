总结 React 生命周期

写在前面

说到React生命周期，就如同人的一生，从出生，到成长，再到凉凉，其中经历的就叫做人的生命周期。
那么 React 也有自己的生命周期。但React更新速度之快，现在版本和之前形成了 “新旧生命周期” ，
现在我们先看一下他们的区别

一、旧的生命周期

Q: React 都已经更新了新的生命周期了，为什么我们还要学习旧的生命周期(钩子)呢

A: 首先 React 在17.x的版本中还没有废弃旧钩子，只是会有弃用警告
其次目前不是所有的项目都用了新的生命周期(钩子)，所以我们旧的也要学
最后新的生命周期(钩子)也只是在旧的生命周期上废弃了三个钩子，增加了两个钩子，其他核心不变

生命周期分为三个阶段：挂载、更新、卸载

我们分两个阶段来研究：挂载和更新 （看上图，无论是挂载还是更新都可以走到卸载）

挂载阶段
1.1 constructor()

constructor() 中完成了React数据的初始化，他接收一个参数：props.
在constructor 中 必须写 super(props);这个是类有关继承的知识点
tips：constructor() 是可有可无的，不写constructor()也不会对我们的页面造成什么影响
在 React 官网中是这么写的

1.2componentWillMount()

这个钩子一般用的比较少，在调用完constructor钩子后，组件还未渲染时调用

1.3 render()

每个组件都要有一个render钩子，这个用来渲染组件，把你想渲染的内容用jsx语法写在这个钩子中

1.4componentDidMount()

在组件完成第一次渲染后调用，如果有页面初始的时候就要执行的事就需要在这个钩子中调用

(例如：在页面一加载完成就启动一个定时器。)

更新阶段
2.1componentWillReceiveProps(props)

组件将要接收新的props 的钩子，这个钩子用于父子组件之间，父组件可以向子组件传递参数，在子组件中写这个钩子，props参数是更新过一次之后的参数

2.2shouldComponentUpdate()

这个钩子可以看成是一个阀门，只有返回值为true的时候，才能继续执行下面的钩子，如果为false，则不执行下面的钩子。
tips: 如果我们在组件中没有写这个钩子，那么React底层会帮我们补全并且返回值为true，如果我们自己在组件中写了这个钩子，那么就一定要有返回值

2.3componentWillUpdate(nextProps，nextState)

在组件更新前调用，一般用的比较少。这里可以拿到nextProps，nextState。看图可以看到这个钩子旁边有个forceUpdate()方法，如果调用这个方法，则会直接触发componentWillUpdate()钩子，这个方法是强制更新，一旦调用这个方法，则会跳过shouldComponentUpdate()这个钩子，直接进行更新。

2.4render()

看图render这个钩子贯穿挂载和更新。所以不管是挂载还是更新都会触发render这个钩子

2.5componentDidUpdate(prevProps,prevState)

除了第一次的挂载，每次更新完后都会调用componentDidUpdate()这个钩子，这里可以拿到prevProps和prevState，即更新前的props和state。

卸载阶段
3.1 componentWillUnmount()

这是组件在卸载前调用的钩子，主要处理组件的卸载和数据的销毁

可以通过这个方法来卸载组件
ReactDOM.unmountComponentAtNode(document.getElementById(id名称));

  1. 清除组件中的setTimeout，setInterval
  2. 移除组件中的监听事件removeEventListener
生命周期(旧)总结

二、新的生命周期

新的生命周期就是多了两个钩子，废除了三个钩子，我们在这里重点讲述一下新增的两个钩子(这两个钩子也不常用)

4.1static getDerivedStateFromProps(props,state)

1.获取来自 Props 的派生状态
2.会在调用 render 方法之前调用，并且在初始挂载及后续更新时都会被调用。
3.这个钩子几乎用不到，适用于state的值完全取决于props
4.它应返回一个对象来更新 state，如果返回 null 则不更新任何内容。
4.2getSnapshotBeforeUpdate(prevProps, prevState)

  1. 在更新之前获取快照，并且传递给 componentDidUpdate 这个钩子
    （可以理解为在更新之前保存好之前的数据，并在更新后把保存的之前的数据传过去）

  2. 在 React 开启异步渲染模式后，在 render 阶段读取到的 DOM 元素状态并不总是和 commit 阶段相同，这就导致在
    componentDidUpdate 中使用 componentWillUpdate 中读取到的 DOM 元素状态是不安全的，因为这时的值很有可能已经失效了。
